## 規約・提出の失敗モード

| 失敗 | 内容 |
|------|------|
| 証拠不一致 | README「Codex 90%」なのに短い相談 Session ID |
| 動画＝一次審査 | 「触れば分かる」は通用しない |
| デモ維持切れ | 提出日だけ動く／クレジット切れ／無料ホス停止 |
| 締切後修正幻想 | 原則変更不可 |
| AI審査の不整合検知 | 説明とコード不一致、カテゴリと実装不一致 |
| 既存薄拡張 | 見た目立派でも評価は薄い新規のみ |
| カテゴリ誤認 | 「教育にも使える」だけで Education |
| 複数応募の焼き直し | 否認され得る |
| Plugin過信 | Official Rules 優先 |
| 利益相反プロジェクト | 過去の優遇支援 |

## 技術・プロダクトの失敗モード

- CRUD + チャット欄だけの GPT-5.6（Stage 1「合理的適用」危険）
- 汎用AI秘書／何でもチャット／巨大プラットフォーム（埋没）
- JSONがターミナルに出るだけ（Dev Tools以外で大減点）
- 機能を10個見せて縦一本が無い動画
- ドメイン証明不能な「暗黙知発見」を審査員に見せる（当事者以外判定不能）
- スクレイピング・因果誤認・権利リスクの高い構想Aを締切内に証明しようとする
- Workspace Agents等に審査環境依存で再現性低下
- 縮小しすぎて「賢い会議プラグイン」になる（`talk8`）
- 「OS全部入り」を約束して未完成で提出

## 対話が明示した確度付き解釈

| 主張 | 確度 | 種別 |
|------|------|------|
| Codex + GPT-5.6 両方必須 | 98% | FAQ寄り |
| 両方を実質使用 | 95% | FAQ寄り |
| GPT-5.6 は完成品機能へ組み込むべき | 90% | **推論** |
| 動画が実質最重要審査資料 | 90% | FAQ寄り推論 |
| 狭い完成品 > 広い未完成 | 85% | 戦略 |
| 単純AIラッパーは上位困難 | 90% | 戦略 |
| Join前repo作成は失格でない | ~90% | 推論 |
| 旧コードの新repoコピーは新規化にならない | ~95% | 推論 |

## 製品構想側で切った危険主張

> 諦めるべきなのは構想ではなく、**外部から検証できない形の主張**。（`talk3`/`talk11`）

提出で証明してはいけないもの:

- 「暗黙知を発見した」
- 「組織の本当のボトルネックを理解した」
- 「人類に良い」（文明論プレゼン）

Living Decisionsを選ぶ場合に証明するもの（talk11枝の案）:

- 決定が出典・前提・権限・異論・実行/再検討条件を持つ Commitment になること
- 後に現実変化で再検討へ遷移すること
