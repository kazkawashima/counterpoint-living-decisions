| 要件 | 内容 | 種別 |
|------|------|------|
| 両方必須 | “Build any project with Codex **and** GPT-5.6”。README・動画でも両方説明 | [FAQ] |
| 既存PJの拡張 | talk10では “Codex **and/or** GPT-5.6” という個別文言も確認。ただし対話の安全解釈は、提出プロジェクト全体では両方を実質利用 | [FAQ] + [推論] + [公式再確認要] |
| 装飾禁止 | incidental / decorative では足りない | [FAQ] |
| Codex | **開発工程**で構築。主要スレッドの `/feedback` Session ID。動画・READMEで関与を説明 | [FAQ] |
| Codex ランタイム | 完成品からの呼び出しは**要求されていない** | [FAQ] |
| GPT-5.6 | プロジェクトに統合し「何をしているか」を動画で説明。開発時のみは危険 | [推論 ~90%] |

### 弱い使い方（対話列挙）

- Codex: READMEだけ、アイデア出しだけ、軽微修正だけ、周辺相談を Session ID に
- GPT-5.6: Codex内部モデルとして使っただけ、名前考えさせただけ、外しても価値が変わらない

### 推奨分担 [戦略]（`talk1`）

```
Codex     = 開発エージェント（Build-time・必須ライン）
GPT-5.6   = 完成品の意味判断・統合機能
通常コード = 決定論処理・状態・ACL・イベント
人間       = 承認境界
```

Runtime Codex は時間があれば上振れ（非必須）[戦略]。

## 審査ステージ

1. **適格性** — 提出物・参加資格・IP・使用ツール
2. **Stage 1（Pass/Fail）** — テーマ適合 + 指定 API/SDK の合理的適用
3. **Stage 2** — 4項目各 **25%**

| 項目 | 見るもの |
|------|---------|
| Technological Implementation | Codexの深さ・巧みさ、動作、非自明実装 |
| Design | 完全で一貫したプロダクト体験（PoCではない） |
| Potential Impact | 実在問題・実在対象・デモが問題を解くか |
| Quality of the Idea | 創造性・新規性・問題領域理解 |

- **タイブレーク順:** Tech → Design → Impact → Idea [FAQ]
- Stage 2 Tech は明示的に Codex 深度を問う。GPT-5.6 深度の独立採点項目はない、というのは対話内の審査文言の読み取り [推論] [公式再確認要]
- 審査員に実テスト義務なし（テキスト／画像／動画のみ可）[FAQ]
